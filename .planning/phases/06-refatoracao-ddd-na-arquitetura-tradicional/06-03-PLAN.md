**Título:** `# Value Objects — Aplicados na Arquitetura Tradicional`

**## O que é** (2-3 parágrafos):
Value Object é um objeto definido pelos seus valores, não por identidade. Dois `PeriodoLetivo(2026, 1)` são o mesmo período — `equals` compara valores, não referências. A propriedade mais importante: imutabilidade. Após criado, um Value Object não muda — se precisar de um valor diferente, cria-se um novo objeto. Para teoria completa, ver `docs/02-design-tatico/value-objects.md`.

**## Manifestação no módulo camadas (ANTES)**:
Usar snippet de `erp-matricula-camadas/src/main/java/br/com/escola/matricula/model/Matricula.java`. Mostrar: `private String periodoInicio` e `private String periodoFim` — strings sem validação. `private UUID alunoId` — UUID sem semântica de domínio. `private String status` — aceita qualquer string. Comentário `// ANTES: primitivos sem validação`. Após o snippet: texto explicando que um `periodoInicio = ""` ou `status = "ACTIVA"` (erro de digitação) passa pela compilação sem nenhuma reclamação. Referência ao arquivo completo.

**## Aplicação no módulo DDD (DEPOIS)**:
Mostrar o record `PeriodoLetivo` com construtor compacto validando `ano >= 2000` e `semestre in {1,2}`. Mostrar também o record `NomeDisciplina` com `strip()`, `isBlank()`, `length() > 100`. Comentário `// DEPOIS: records imutáveis com validação no construtor`. Após o snippet: ponto pedagógico central — "um `PeriodoLetivo(2026, 3)` é impossível de criar — o construtor lança `IllegalArgumentException` antes que o objeto exista. Com `String periodoFim`, um semestre 3 passa pela compilação e só falha em runtime, horas depois."

**## O que foi ganho**:
3 linhas: erros de dados impossíveis em tempo de compilação (se o tipo existe, é válido). Sem validação duplicada em múltiplos Services — a validação fica no construtor do record. Records Java 21 são naturalmente imutáveis (sem setters).

Regras críticas:

- Confirmar o conteúdo exato dos arquivos VO antes de escrever — particularmente PeriodoLetivo.java e NomeDisciplina.java
- Não ultrapassar 15 linhas por snippet

**Título:** `# Agregados — Aplicados na Arquitetura Tradicional`

**## O que é** (2-3 parágrafos):
Aggregate é um cluster de objetos tratado como unidade de consistência. O Aggregate Root controla o acesso — toda modificação passa pelos métodos do Root, nunca diretamente nos objetos internos. A principal garantia: as invariantes do grupo são protegidas pelo próprio Aggregate, não por qualquer Service externo. Para aprofundamento, ver `docs/02-design-tatico/agregados.md`.

**## Manifestação no módulo camadas (ANTES)**:
Snippet de `MatriculaServiceImpl.adicionarDisciplina()` mostrando: verificação de status como `if (!"ATIVA".equals(matricula.getStatus()))` (Service interpreta String), e a query `itemMatriculaRepository.countByMatriculaId(matriculaId)` seguida de `if (qtd >= 6)`. Comentário `// ANTES: regras espalhadas no Service`. Nota: a referência correta é `itemMatriculaRepository.countByMatriculaId` — não `matriculaRepository.countDisciplinas` que é dead code.

Após o snippet, texto explicando o problema de concorrência: "dois usuários simultâneos podem cada um ler 5 disciplinas e adicionar — resultado: 7 disciplinas. A verificação e a adição são operações separadas no Service."

**## Aplicação no módulo DDD (DEPOIS)**:
Snippet de `Matricula.adicionarDisciplina()` mostrando os 3 guards em sequência: Guard 1 com `instanceof StatusMatricula.Cancelada`, Guard 2 com `this.disciplinas.size() >= LIMITE_DISCIPLINAS`, Guard 3 com `stream().anyMatch(...)`. Comentário `// DEPOIS: invariantes protegidas pelo Aggregate`.

**## O que foi ganho**:
3 linhas: verificação e modificação no mesmo objeto, na mesma transação — sem janela de concorrência. Regras não podem ser puladas por chamador externo: sem `setStatus("ATIVA")` público. Exceções específicas de domínio em vez de `RuntimeException` genérica.

**Título:** `# Repositórios — Aplicados na Arquitetura Tradicional`

**## O que é** (2-3 parágrafos):
Repositório é a abstração que recupera e persiste Aggregates. O ponto DDD: a interface do Repositório vive no domínio — não na infraestrutura. Isso inverte a dependência: a infraestrutura implementa a interface do domínio, não o contrário. Diferença de um DAO genérico: o Repositório fala a linguagem do domínio (buscarPorAluno, existeMatriculaAtiva), não a linguagem do banco (findBy, countWhere). Para teoria, ver `docs/02-design-tatico/repositorios.md`.

**## Manifestação no módulo camadas (ANTES)**:
Snippet de `erp-matricula-camadas/src/main/java/br/com/escola/matricula/repository/MatriculaRepository.java` mostrando: `@Mapper` annotation, `findById(UUID id)`, `void save(Matricula matricula)`, e qualquer método que expose SQL como comportamento (ex: countDisciplinas). Comentário `// ANTES: DAO com anotação de infraestrutura e termos técnicos`.

**## Aplicação no módulo DDD (DEPOIS)**:
Snippet de `erp-matricula-ddd/src/main/java/br/com/escola/matricula/dominio/repositorio/MatriculaRepositorio.java` mostrando: ausência de imports de framework (comentário explícito), `buscarPorId(MatriculaId id)`, `buscarPorAluno(AlunoId alunoId)`, `existeMatriculaAtiva(AlunoId, PeriodoLetivo)`, `salvar(Matricula)`. Comentário `// DEPOIS: interface no domínio, sem imports de infraestrutura`.

Adicionar nota: "A implementação `MatriculaRepositorioMyBatis` está em `infraestrutura/persistencia/` — a seta de dependência vai de `infraestrutura/` → `dominio/`, nunca o contrário."

**## O que foi ganho**:
3 linhas: UseCases testáveis sem banco real — basta implementar a interface com um Map em memória. Interface expressa perguntas de negócio (existeMatriculaAtiva?), não operações de banco (count). O domínio não sabe nada sobre MyBatis, PostgreSQL, ou qualquer tecnologia de persistência.
