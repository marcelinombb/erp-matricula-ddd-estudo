CRIAR docs/04-material-didatico/ddd-vs-camadas.md (entrega DID-01):

Título: "DDD para quem vem da Arquitetura em Camadas"
Subtítulo/introdução: explicar que este documento mostra como cada artefato familiar de arquitetura em camadas tem um equivalente DDD no código, e que DDD não é apenas nomenclatura diferente — é responsabilidade diferente.

Organizar em seções comparativas. Para cada seção: coluna "Arquitetura em Camadas" com padrão convencional e coluna "DDD (neste projeto)" com o equivalente e referência ao arquivo real. Exemplos concretos usando o fluxo de matrícula:

1. SEÇÃO: "De @Service para UseCase + Domain Service"
   - Lado tradicional: @Service MatriculaService com método matricular() que valida, persiste, publica — tudo junto
   - Lado DDD: MatricularAlunoUseCase (aplicacao/) orquestra sem decidir + VerificadorElegibilidadeMatricula (dominio/servico/) valida + Matricula.adicionarDisciplina() encapsula invariantes
   - Referências: erp-matricula-app/src/main/java/br/com/escola/matricula/aplicacao/MatricularAlunoUseCase.java e dominio/servico/VerificadorElegibilidadeMatricula.java
   - Por que importa: "O Service tradicional cresce e vira God Object. O UseCase orquestra e o Domain Service encapsula regras — papéis separados, responsabilidades claras."

2. SEÇÃO: "De @Entity para Aggregate Root sem anotações"
   - Lado tradicional: @Entity @Table @Column class Matricula com campos primitivos
   - Lado DDD: class Matricula sem nenhuma anotação de persistência, com métodos adicionarDisciplina() e cancelar() que protegem invariantes
   - Referência: dominio/modelo/Matricula.java
   - Por que importa: demonstra que o domínio não conhece MyBatis — ADR-001

3. SEÇÃO: "De @Repository extends JpaRepository para Interface de Domínio + Implementação Separada"
   - Lado tradicional: MatriculaRepository extends JpaRepository — framework define a interface
   - Lado DDD: MatriculaRepositorio (dominio/repositorio/) define a interface (sem imports); MatriculaRepositorioMyBatis (infraestrutura/persistencia/) implementa
   - Princípio de Inversão de Dependência: o domínio define o contrato, a infraestrutura obedece
   - Por que importa: "A interface de repositório pertence ao domínio porque os casos de uso a usam — não à infraestrutura que a implementa."

4. SEÇÃO: "De String status para Sealed Interface com Pattern Matching"
   - Lado tradicional: String status = "ATIVA"; if (status.equals("CANCELADA")) — "magic strings"
   - Lado DDD: sealed interface StatusMatricula com Ativa, Cancelada, Concluida; switch exaustivo sem default
   - Referência: dominio/modelo/StatusMatricula.java e MatriculaDto.java (switch no factory method)
   - Por que importa: "Compilador detecta estados não tratados — segurança em tempo de compilação."

5. SEÇÃO: "De RuntimeException genérica para Exceções Tipadas com Dados Estruturados"
   - Lado tradicional: throw new RuntimeException("Limite excedido") — sem dados contextuais
   - Lado DDD: throw new LimiteDisciplinasExcedidoException(limite, atual, matriculaId) — dados estruturados que o ExcecaoHandler usa para 422 com campos "limite" e "atual"
   - Referência: dominio/excecao/LimiteDisciplinasExcedidoException.java e interfaces/ExcecaoHandler.java

Fechar com parágrafo curto: "DDD adiciona estrutura, não burocracia. Cada camada tem um propósito claro. O resultado é um código que comunica sua intenção."

CRIAR docs/04-material-didatico/licoes-aprendidas.md (entrega DID-07):

Título: "Lições Aprendidas — DDD na Prática"
Introdução: "Este documento não é marketing de DDD. É uma análise honesta do que funcionou, do que custou mais do que esperávamos, e do que faríamos diferente."

SEÇÃO 1: "O que DDD resolveu bem"
Exemplos concretos com referências a arquivos:
  - "Invariantes do Aggregate": Matricula.adicionarDisciplina() — regra de limite de disciplinas nunca está em um Service, nunca é duplicada. Em uma arquitetura tradicional, essa verificação provavelmente estaria em MatriculaService, possivelmente duplicada em outro ponto.
  - "Exceções com semântica": LimiteDisciplinasExcedidoException com getLimite() e getAtual() — o ExcecaoHandler usa esses dados para um 422 com payload estruturado. RuntimeException genérica perderia esses dados.
  - "Separação domínio/persistência": MatriculaRowMapper.java é o único arquivo que "traduz" entre mundo relacional e mundo de domínio. Adicionar um campo ao banco é cirúrgico — só muda o Row, o Mapper, e o XML.
  - "Domain Events para integração sem acoplamento": FinanceiroEventListener e AcademicoEventListener recebem AlunoMatriculado sem que Matricula conheça Financeiro. A ordem de execução (@TransactionalEventListener) garante que os listeners só são chamados após commit.

SEÇÃO 2: "O que custou mais do que o esperado"
Trade-offs honestos:
  - "Construção de objetos de domínio no Controller": o Controller precisa construir Aluno e Turma completos com dados mínimos (apenas IDs) porque MatricularAlunoCommand recebe objetos de domínio. Em um projeto maior, seria necessário um passo de busca no repositório ou um Command mais simples com IDs primitivos.
  - "MyBatis é mais verboso que JPA": cada campo do banco precisa ser mapeado explicitamente no XML. O ganho pedagógico é alto, mas o custo de manutenção cresce com o número de campos.
  - "Sem Lombok significa mais boilerplate": construtores, getters e equals/hashCode escritos à mão. Java 21 records eliminam o problema para Value Objects e Commands, mas Entidades ainda requerem código manual.
  - "Aggregate boundary é difícil de acertar de primeira": a decisão de não incluir vagas disponíveis no Aggregate Matrícula (STATE.md Decisions) foi tomada deliberadamente para manter o foco. Em produção, essa decisão requereria análise mais profunda dos invariantes cross-Aggregate.

SEÇÃO 3: "O que faríamos diferente em produção"
  - Optimistic locking (coluna version) para concorrência em MatriculaRepositorioMyBatis — documentado como v2 requirement PROD-01
  - Commands com IDs primitivos em vez de objetos de domínio completos — simplificaria o Controller
  - Testes unitários do Aggregate antes de implementar (TDD) — garantiria invariantes sem depender do DemoRunner
  - Separar leitura de escrita explicitamente (CQRS simples) — MatriculaDto de leitura direta via Mapper sem reconstituir o Aggregate

Fechar com: "DDD agrega valor em domínios ricos em comportamento e complexidade de negócio. Para CRUD simples, a estrutura adicional não se paga. Matrícula escolar é um bom caso de uso: regras de negócio claras, estados bem definidos, integração entre contextos."
