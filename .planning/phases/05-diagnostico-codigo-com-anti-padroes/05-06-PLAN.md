    H2: Relação com outros anti-padrões — Service Anêmico e Service Deus (DIAG-03) coexistem: o Service está inchado exatamente porque o modelo não tem comportamento.

    Criar docs/00-ddd-sem-mudar-arquitetura/03-service-deus.md:
    H1: "Anti-padrão: Service Deus"
    H2: O que é — uma classe que cresce sem parar. Todo novo requisito de negócio relacionado à matrícula termina aqui.
    H2: Manifestação — listar os métodos de MatriculaServiceImpl: matricular (60 linhas), adicionarDisciplina (50 linhas), cancelar (30 linhas), buscarPorAluno (15 linhas), verificarElegibilidade (15 linhas). Total: 200+ linhas em uma única classe.
    H2: Por que acontece — relação causal com DIAG-01 e DIAG-02: modelo anêmico força toda lógica para o Service. Cada nova feature adiciona métodos.
    H2: Contraste DDD — no módulo DDD, MatricularAlunoUseCase tem ~109 linhas e só orquestra. CancelarMatriculaUseCase, AdicionarDisciplinaUseCase são classes separadas.
    H2: Consequência prática — quando dois desenvolvedores trabalham em features simultâneas de matrícula, ambos modificam o mesmo Service. Conflitos de merge frequentes.

    Criar docs/00-ddd-sem-mudar-arquitetura/04-duplicacao-regras.md:
    H1: "Anti-padrão: Duplicação de Regras"
    H2: O que é — a mesma regra de negócio implementada em mais de um lugar. Com o tempo, as implementações divergem.
    H2: Manifestação — mostrar que if (!aluno.isAtivo()) aparece tanto em MatriculaServiceImpl.matricular() quanto em DisciplinaServiceImpl.adicionarDisciplinaExtra(). Citar ambos os arquivos.
    H2: Como a divergência acontece — narrar o cenário: 6 meses após o código ser criado, um desenvolvedor adiciona exceção para "alunos em período de teste" em MatriculaServiceImpl. DisciplinaServiceImpl não é atualizado. Agora um aluno em período de teste pode adicionar disciplinas mas não pode se matricular — comportamento incoerente.
    H2: Por que acontece — sem modelo de domínio, regras vivem no Service. Quando um segundo Service precisa da mesma regra, copia-se o if.
    H2: Contraste DDD — no módulo DDD, a regra de elegibilidade vive em VerificadorElegibilidadeMatricula. Todos os UseCases que precisam verificar elegibilidade chamam o mesmo Domain Service.

    Criar docs/00-ddd-sem-mudar-arquitetura/05-regras-na-interface.md:
    H1: "Anti-padrão: Regras na Interface"
    H2: O que é — validações de regra de negócio executadas apenas no Controller HTTP. A regra depende do protocolo para existir.
    H2: Manifestação — snippet de MatriculaController.matricular() mostrando a validação de período que existe APENAS no Controller. Citar arquivo.
    H2: Quando a regra desaparece — listar os cenários onde a validação é bypassada: batch job que chama MatriculaService.matricular() diretamente; teste de integração; chamada interna via ApplicationContext.
    H2: Sutil mas perigoso — esta é a mais difícil de detectar: o código funciona nos testes de contrato HTTP. Só falha quando o Service é usado fora do contexto HTTP.
    H2: Contraste DDD — no módulo DDD, o Controller não contém regras de negócio. Todo o comportamento está nos UseCases e no modelo de domínio.

    Criar docs/00-ddd-sem-mudar-arquitetura/06-acoplamento-banco.md:
    H1: "Anti-padrão: Acoplamento ao Banco"
    H2: O que é — regras de negócio modeladas em função das tabelas, não do negócio. Para encontrar uma regra, você precisa rastrear até o SQL.
    H2: Manifestação 1 — Matricula.java no módulo camadas: campos nomeados igual às colunas (alunoId, turmaId, periodoInicio). String status em vez de tipo de domínio.
    H2: Manifestação 2 — a regra "máximo 6 disciplinas" não existe em Matricula.java. Existe como SELECT COUNT(*) em itemMatriculaRepository.countByMatriculaId() chamado em MatriculaServiceImpl. Citar arquivo e método.
    H2: Problema de descoberta — um desenvolvedor novo lê Matricula.java e não encontra o limite de 6 disciplinas. Precisa ler MatriculaServiceImpl, rastrear até countByMatriculaId, e ler o SQL no XML do mapper. Três arquivos para encontrar uma regra.
    H2: Contraste DDD — no módulo DDD, Matricula.adicionarDisciplina() lança exceção quando há 6 itens em getItens(). A regra vive no modelo. Um desenvolvedor lê Matricula.java e encontra a regra.
    H2: Nota — o anti-padrão não é usar SQL. É usar SQL como repositório de regras de negócio. MyBatis com SQL explícito é totalmente compatível com DDD quando o modelo de domínio protege suas invariantes independentemente do banco.
