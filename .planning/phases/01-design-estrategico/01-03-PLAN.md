    --- ADR-001-mybatis-vs-jpa.md ---

    Cabeçalho:
    ```
    # ADR-001: MyBatis em vez de JPA/Hibernate

    **Status:** Aceito
    **Data:** 2026-06-20
    **Contexto da fase:** Fase 1 — Design Estratégico
    ```

    "## Contexto"
    Em projetos Java com banco relacional, a escolha padrão é JPA/Hibernate via Spring Data JPA. Este projeto deliberadamente rejeita essa escolha — a motivação precisa ser explícita.

    O problema com JPA em projetos DDD: anotações de persistência entram nas classes de domínio. Incluir o bloco de código "COM JPA" seguido do bloco "COM MYBATIS":

    Bloco COM JPA (problema):
    ```java
    // COM JPA — anotações de persistência no modelo de domínio (PROBLEMA)
    @Entity
    @Table(name = "matriculas")
    public class Matricula {
        @Id
        @GeneratedValue(strategy = GenerationType.UUID)
        private UUID id;

        @OneToMany(cascade = CascadeType.ALL, fetch = FetchType.LAZY)
        @JoinColumn(name = "matricula_id")
        private List<ItemMatricula> itens;

        // O domínio agora importa jakarta.persistence.* — depende do framework
    }
    ```

    Bloco COM MYBATIS (solução):
    ```java
    // COM MYBATIS — modelo de domínio limpo (DECISÃO TOMADA)
    public class Matricula {
        private final MatriculaId id;
        private final List<ItemMatricula> itens;

        // Zero imports de framework — puro Java
        // A conversão para/de banco ocorre em MatriculaRowMapper (infraestrutura/)
    }
    ```

    Após os blocos, um parágrafo explicando: o modelo de domínio com JPA importa `jakarta.persistence.*` — isso viola o princípio DDD de que o domínio não conhece infraestrutura. O desenvolvedor vindo de Spring Boot não percebe o problema até ver o import explicitamente.

    "## Alternativas Consideradas"
    Subseções para: JPA/Hibernate via Spring Data JPA (com prós e contras), Spring Data JDBC (mais próximo de DDD que JPA, mas ainda impõe `@Table`/`@Id`), JDBC puro sem framework (mais explícito, mas mais verboso sem ganho didático adicional sobre MyBatis).

    "## Decisão"
    Usamos MyBatis (mybatis-spring-boot-starter 3.0.5). O mapeamento SQL é explícito em arquivos .xml. A conversão entre modelo relacional e modelo de domínio ocorre em classes `*RowMapper` dentro do pacote `infraestrutura/`.

    "## Consequências"
    Positivas: domínio sem imports de framework (verificável via grep), conversão explícita e legível em RowMapper, mapeamento de Aggregate complexo com JOIN + collection sem magic, pedagogia clara da separação domínio/persistência.
    Negativas (Trade-offs): mais XML e código de mapeamento escrito manualmente, sem auto-CRUD (cada query é escrita explicitamente), curva de aprendizado do XML MyBatis para desenvolvedores habituados ao Spring Data.

    "## Referências"
    - MyBatis Result Maps: https://mybatis.org/mybatis-3/sqlmap-xml.html
    - Separating Persistence and Domain Models: https://urgo.medium.com/separating-persistence-and-domain-models-cc3a7e7cd4e5

    --- ADR-002-escopo-bounded-context.md ---

    Cabeçalho:
    ```
    # ADR-002: Implementar Apenas o Bounded Context de Matrícula

    **Status:** Aceito
    **Data:** 2026-06-20
    **Contexto da fase:** Fase 1 — Design Estratégico
    ```

    "## Contexto"
    Um ERP escolar real teria múltiplos Bounded Contexts implementados (Matrícula, Financeiro, Acadêmico, Secretaria). Este projeto implementa apenas o BC Matrícula, deixando Financeiro e Acadêmico como stubs. Esta decisão precisa ser justificada — do contrário, parece uma omissão.

    O problema de implementar múltiplos BCs num projeto didático: integração real entre contextos exige infraestrutura de messaging (Kafka, RabbitMQ) ou contratos de API, e o desenvolvedor ainda não consolidou os padrões táticos do Core Domain. Adicionar complexidade de integração real antes que o domínio esteja claro é a sequência errada de aprendizado.

    "## Alternativas Consideradas"
    Implementar Financeiro parcialmente como consumidor real de AlunoMatriculado (mais realista, mais complexo), implementar via REST entre BCs (adiciona complexidade HTTP antes do domínio estar consolidado), implementar Financeiro e Acadêmico completos (desvio de foco — os padrões táticos são os mesmos que Matrícula já demonstra).

    "## Decisão"
    Apenas o BC Matrícula está implementado. Financeiro e Acadêmico existem como listeners stub com `@TransactionalEventListener` que demonstram o contrato de integração sem implementar a lógica real dos contextos downstream.

    "## Consequências"
    Positivas: foco total nos padrões táticos do Core Domain, sem overhead de infraestrutura de mensageria, código de stub demonstra o padrão de integração via eventos sem poluição.
    Negativas (Trade-offs): o desenvolvedor não vê consistência eventual real entre contextos, não experimenta o desafio de manter contratos de evento entre equipes, integração in-process com Spring ApplicationEvents é diferente de messaging real (Kafka/RabbitMQ) em produção.

    "## Referências"
    - Bounded Context — Martin Fowler: https://martinfowler.com/bliki/BoundedContext.html

    --- ADR-003-referencia-por-id.md ---

    Cabeçalho:
    ```
    # ADR-003: Referência Entre Aggregates por ID, Não por Objeto

    **Status:** Aceito
    **Data:** 2026-06-20
    **Contexto da fase:** Fase 1 — Design Estratégico
    ```

    "## Contexto"
    Em DDD, a regra é que Aggregates de BCs diferentes não se referenciam por objeto Java — apenas por ID. Esta decisão parece técnica, mas tem consequências de design profundas. Sem essa regra, é tentador carregar o objeto completo "para facilitar":

    Bloco EVITAR (problema):
    ```java
    // EVITAR: referência por objeto entre Aggregates
    public class Matricula {
        private Aluno aluno;          // carrega o Aggregate Aluno inteiro
        private Turma turma;          // carrega o Aggregate Turma inteiro
        private PeriodoLetivo periodo;
    }
    ```

    Bloco PREFERIR (decisão):
    ```java
    // PREFERIR: referência por ID entre Aggregates
    public class Matricula {
        private AlunoId alunoId;      // apenas o identificador
        private TurmaId turmaId;      // apenas o identificador
        private PeriodoLetivo periodo; // Value Object: faz parte do Aggregate
    }
    ```

    Parágrafo explicando: com referência por objeto, qualquer operação em Matrícula carrega `Aluno` completo (com todos os dados que o BC Financeiro ou Acadêmico precisam) — mesmo quando só precisamos saber se o aluno está ativo. Isso cria acoplamento entre Aggregates de BCs distintos e viola o princípio de que cada BC tem seu próprio modelo.

    Parágrafo sobre banco: no schema SQL, isso significa `aluno_id UUID` na tabela `matricula` sem `FOREIGN KEY REFERENCES alunos(id)`. A ausência de FK entre tabelas de BCs diferentes é intencional — os BCs podem estar em bancos distintos no futuro. A consistência é garantida por Domain Events, não por FK relacional.

    "## Alternativas Consideradas"
    Referência por objeto Aluno completo (mais conveniente no código, acopla BCs), FOREIGN KEY relacional entre tabelas de BCs distintos (conveniente no banco, cria acoplamento de schema), referência por ID apenas para BCs externos, objetos para entidades do mesmo BC (decisão intermediária adotada — PeriodoLetivo é VO interno, AlunoId e TurmaId são referências externas).

    "## Decisão"
    Aggregates de Bounded Contexts diferentes são referenciados apenas por ID tipado (`AlunoId`, `TurmaId`). Esses IDs são records Java (Value Objects), não UUIDs crus — evitam confusão entre IDs de tipos diferentes em tempo de compilação. Entidades internas ao mesmo Aggregate (como `ItemMatricula` dentro de `Matricula`) são referenciadas por objeto.

    "## Consequências"
    Positivas: Aggregates menores e focados, sem carga desnecessária de dados, possibilidade de BCs em bancos distintos no futuro, detecção em tempo de compilação de confusão entre IDs (`AlunoId` não é `TurmaId`).
    Negativas (Trade-offs): Application Service precisa carregar `Aluno` e `Matricula` separadamente quando ambos são necessários — mais código de orquestração. Sem FK relacional, a consistência entre tabelas de BCs distintos depende da lógica da aplicação e dos Domain Events.

    "## Referências"
    - Consistency Boundary — James Hickey: https://www.jamesmichaelhickey.com/consistency-boundary/
    - contexto-matricula.md §15 (Relacionamentos Entre Aggregates)

    --- ADR-004-codigo-em-portugues.md ---

    Cabeçalho:
    ```
    # ADR-004: Código, SQL e Arquivos em Português

    **Status:** Aceito
    **Data:** 2026-06-20
    **Contexto da fase:** Fase 1 — Design Estratégico
    ```

    "## Contexto"
    A convenção universal em projetos Java é nomear classes, métodos e variáveis em inglês. Este projeto faz o oposto deliberadamente. A motivação é pedagógica e DDD: a Linguagem Ubíqua não é só documentação — ela vive no código.

    Bloco de contraste:
    ```java
    // INGLÊS TÉCNICO (padrão de mercado)
    matriculaRepository.findById(id);
    enrollment.addSubject(subject);
    student.isActive();

    // PORTUGUÊS (decisão deste projeto)
    matriculaRepositorio.buscarPorId(id);
    matricula.adicionarDisciplina(disciplina);
    aluno.estaAtivo();
    ```

    Parágrafo: no segundo exemplo, o especialista de negócio consegue ler o código e reconhecer os conceitos do seu domínio. "buscarPorId" é o mesmo verbo que ele usa ao falar sobre a operação. "adicionarDisciplina" é exatamente o que acontece no processo de negócio. A tradução de inglês para português — mesmo que inconsciente — cria uma barreira cognitiva entre a fala do domínio e o código.

    "## Alternativas Consideradas"
    Inglês para tudo (padrão de mercado, ferramentas otimizadas, Stack Overflow funciona melhor), português para documentação e inglês para código (elimina a barreira cognitiva só parcialmente — o código ainda usa termos diferentes do negócio), português apenas para nomes de domínio e inglês para infraestrutura (complexidade de regra — difícil de aplicar consistentemente).

    "## Decisão"
    Todos os identificadores de código Java (`class`, `interface`, `record`, nomes de método, nomes de variável), nomes de tabela e coluna SQL, e nomes de arquivo de configuração usam português. Exceções permitidas: palavras-chave da linguagem (`public`, `class`, `void`), anotações de framework (`@Entity`, `@Service`), identificadores de biblioteca externa.

    "## Consequências"
    Positivas: código como Linguagem Ubíqua — sem tradução entre fala do negócio e código, especialistas de domínio conseguem revisar regras de negócio no código, reforça o princípio DDD a cada linha escrita.
    Negativas (Trade-offs): ferramentas de análise estática, AI assistants e Stack Overflow funcionam melhor com inglês — o desenvolvedor paga esse custo em produtividade de ferramentas, código em português é incomum no mercado e pode causar estranhamento em code reviews com pessoas externas ao projeto, algumas IDEs e plugins assumem nomenclatura em inglês para sugestões de nome.

    "## Referências"
    - Domain-Driven Design — Eric Evans (Ubiquitous Language)
    - contexto-matricula.md §2 (Ubiquitous Language)
