**Abertura bottom-up (D-04):** NÃO começa com "Em DDD, uma Entidade é...". Começa com o contraste concreto: "Vimos que `Cpf` é imutável — dois CPFs com o mesmo valor são o mesmo CPF. Dois ou três parágrafos antes de introduzir o nome "Entidade".

**Tabela de contraste (padrão de Conceitos Ambíguos de linguagem-ubiqua.md):** Logo após a abertura, tabela:

| Característica | Value Object (ex: `Cpf`) | Entidade (ex: `Aluno`) |
|---------------|--------------------------|------------------------|
| Identidade | Sem identidade — comparado por valor | Tem identidade — `AlunoId` persiste |
| Imutabilidade | Imutável — `record` Java 21 | Pode mudar estado (status ativo/inativo) |
| `equals`/`hashCode` | Por todos os campos (gerado pelo `record`) | Apenas pelo `id` — implementado manualmente |
| Tipo Java 21 | `record` — imutabilidade garantida pelo compilador | `class` — permite estado mutável |

**Subseções por Entidade (D-07, D-08):** Duas subseções `###`: `### Aluno` e `### Turma`. Cada uma com sub-headers "Identidade", "Ciclo de Vida", "Responsabilidades" (padrão de bounded-contexts.md adaptado).

Na subseção `### Aluno`:
- Snippet Java 21 completo conforme RESEARCH.md TAT-01: `record AlunoId(UUID valor)` com comentário Java 21/DDD fit, depois classe `Aluno` com `final AlunoId id`, `boolean ativo`, construtor, `estaAtivo()`, `equals`/`hashCode` por id apenas.
- Comentário explícito: `// NÃO é um record porque status pode mudar — record é imutável` e `// DDD fit: Entidade = identidade estável + estado mutável`.
- Blockquote: `> **Por que não é um record?** `Aluno` pode mudar de status (ativo → inativo). Um `record` Java 21 é imutável por design — campos `final`, sem setters. Entidades têm ciclo de vida com estado mutável; para isso usamos `class`.`

Na subseção `### Turma`:
- Snippet mais simples: `record TurmaId(UUID valor)`, classe `Turma` com `TurmaId id`, `int capacidadeMaxima`, `PeriodoLetivo periodo`.
- Comentário sobre `Turma` no v1: sem Aggregate próprio — vagas disponíveis não são verificadas no v1 (conforme STATE.md open decision). Nota inline: "(verificação de vagas disponíveis: v2 — ver STATE.md open decisions)".
- Cross-reference: `Turma` referenciada por `TurmaId` no Aggregate `Matricula` — ver [ADR-003](../adrs/ADR-003-referencia-por-id.md).

**Nota sobre ItemMatricula:** Após as duas subseções, parágrafo explícito: "Existe uma terceira entidade, `ItemMatricula`, que representa uma disciplina incluída na matrícula. Ela não é documentada aqui porque não tem significado fora do contexto do Aggregate `Matricula` — é uma entidade interna. Sua documentação está em [agregados.md](./agregados.md)."

**Seção "## Erros Comuns" (D-05, D-09):** Ao menos dois pares ERRADO/CERTO:

1. Erro "equals por conteúdo" — ERRADO mostra `@Override boolean equals(Object o)` comparando `nome`, `cpf`, etc. (como se fosse VO). CERTO mostra comparação apenas por `this.id.equals(outro.id)`. Comentário: "dois alunos podem ter o mesmo nome — o que os distingue é o ID".

**Nomenclatura:** todos os identificadores em português. `@Override` é permitido — é anotação da linguagem Java, não de framework.
