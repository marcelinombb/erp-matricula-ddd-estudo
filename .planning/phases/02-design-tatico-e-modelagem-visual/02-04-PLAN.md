**DIAGRAMA 1 — MOD-01: Diagrama de Classes**

Texto introdutório (2-3 linhas, D-12): "O diagrama de classes mostra todos os elementos do domínio de Matrícula e seus relacionamentos. Observe que `Matricula` referencia `AlunoId` e `TurmaId` — não os objetos `Aluno` e `Turma` — implementando [ADR-003](../adrs/ADR-003-referencia-por-id.md). `StatusMatricula` é uma `sealed interface` com três implementações como `record` — ver [agregados.md](./agregados.md)."

Bloco Mermaid classDiagram EXATO conforme RESEARCH.md §MOD-01. Atenção aos pitfalls:
- Usar `List~ItemMatricula~` (tilde, não ângulo) — pitfall crítico do RESEARCH.md
- `<<sealed interface>>`, `<<record>>`, `<<interface>>` são os stereotypes corretos
- Incluir `%%` comments para organizar seções (Aggregate Root, Value Objects, Entidades, Repository)
- Relações: `*--` composição, `-->` associação, `<|--` implementação, `..>` dependência

Blockquote após o diagrama: `> **O que observar:** As linhas `-->` de `Matricula` para `AlunoId` e `TurmaId` — não para `Aluno` e `Turma`. Esse é o padrão de referência por ID (ADR-003) visualizado. Domain Events não aparecem aqui — estão no sequence diagram.`

Blockquote após o diagrama: `> **Limite de consistência:** Tudo dentro do `subgraph` é carregado e salvo atomicamente pelo `MatriculaRepositorio`. `Aluno` e `Turma` são Aggregates separados — cada um com seu próprio Repositório (que não é implementado no v1).`

**DIAGRAMA 3 — MOD-03: Flowcharts de Negócio (3 fluxos)**

Texto introdutório (2-3 linhas, D-12): "Os três fluxos de negócio mostram o caminho feliz e os caminhos de erro para cada caso de uso. Os diamantes `{}` representam as invariantes do Aggregate verificadas em `adicionarDisciplina()`, `cancelar()` e `Matricula.criar()` — ver [agregados.md](./agregados.md). Cada exceção nomeada mapeia para uma exceção de domínio tipada."

Três blocos Mermaid flowchart TD conforme RESEARCH.md §MOD-03, sequencialmente:
1. `### Fluxo 1: Realizar Matrícula` — com nós de decisão para Aluno ativo, Período aberto, Matrícula duplicada.
2. `### Fluxo 2: Adicionar Disciplina` — com nós de decisão para cancelada, duplicada, limite.
3. `### Fluxo 3: Cancelar Matrícula` — com nós de decisão para inexistente, já cancelada.

Atenção crítica ao pitfall de Mermaid: toda aresta saindo de nó de decisão `{...}` deve ter label `-->|Sim|` e `-->|Não|` (ou equivalente). Sem labels = diagrama ambíguo.

Blockquote após os três fluxos: `> **Nota:** Os nomes de exceção nos fluxos (`AlunoInativoException`, `LimiteDisciplinasExcedidoException`, etc.) correspondem às exceções tipadas documentadas em [agregados.md](./agregados.md). Em HTTP, elas resultam em respostas 422 (violação de invariante) ou 409 (conflito) — mapeamento detalhado na Fase 4.`
