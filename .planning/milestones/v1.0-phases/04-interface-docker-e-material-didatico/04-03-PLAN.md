Ambiente do host:
  - Java 17 instalado (não Java 21) — Java 21 só disponível via Docker
  - Maven NÃO instalado no host
  - Docker 29.6.0 disponível
  - Docker Compose v5.1.4 disponível

Pitfall crítico (RESEARCH.md Pitfall 4): mvnw ausente no repositório.
Solução: instalar Maven no stage de build via "apk add --no-cache maven".

Pitfall crítico (RESEARCH.md Pitfall 3): depends_on sem condition não garante banco pronto.
Solução: healthcheck com pg_isready + depends_on: condition: service_healthy.

Comando Maven para build single-module:
  mvn -q package -DskipTests -f erp-matricula-app/pom.xml
  OU: mvn -q package -DskipTests -pl erp-matricula-app
  JAR gerado em: erp-matricula-app/target/erp-matricula-app-*.jar

Credenciais (per D-11):
  POSTGRES_USER: ${DB_USER:-matricula}
  POSTGRES_PASSWORD: ${DB_PASSWORD:-matricula}
  POSTGRES_DB: erp_matricula

Spring Boot env var override (per D-11, RESEARCH.md "Don't Hand-Roll"):
  SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/erp_matricula
  SPRING_DATASOURCE_USERNAME=${DB_USER:-matricula}
  SPRING_DATASOURCE_PASSWORD=${DB_PASSWORD:-matricula}

STAGE 1 — builder:
  FROM eclipse-temurin:21-jdk-alpine AS builder
  WORKDIR /build
  Instalar Maven: RUN apk add --no-cache maven
  Copiar o projeto inteiro: COPY . .
  Compilar apenas o módulo da aplicação: RUN mvn -q package -DskipTests -f erp-matricula-app/pom.xml
  Comentário inline: "# -DskipTests: testes automatizados são v2 (TEST-01..03 no REQUIREMENTS.md)"

STAGE 2 — runtime:
  FROM eclipse-temurin:21-jre-alpine
  WORKDIR /app
  Copiar apenas o JAR do stage anterior: COPY --from=builder /build/erp-matricula-app/target/erp-matricula-app-*.jar app.jar
  Nota: se o wildcard não funcionar com COPY, verificar o nome exato no pom.xml e usar o nome literal.
  ENTRYPOINT ["java", "-jar", "app.jar"]

Comentários pedagógicos no Dockerfile:
  - Acima do FROM builder: "# Stage 1: Compilação — JDK completo para compilar com Maven"
  - Acima do apk add: "# Maven instalado no container (mvnw não está no repositório)"
  - Acima do FROM runtime: "# Stage 2: Runtime — apenas JRE (menor que JDK) + o JAR compilado"
  - Explicar o conceito de multi-stage como boa prática: "# A imagem final não contém Maven, JDK ou código-fonte — apenas o JAR executável"

HEALTHCHECK de aplicação (opcional mas pedagógico): adicionar HEALTHCHECK CMD wget -q --spider http://localhost:8080/actuator/health || exit 1 — ou omitir se actuator não estiver disponível. Verificar se spring-boot-starter-actuator está no pom.xml antes de adicionar. Se não estiver, omitir o HEALTHCHECK do Dockerfile (o healthcheck do postgres é suficiente no compose).

Versão do compose: não declarar a chave "version" obsoleta (Docker Compose v2+ não requer).

SERVICE postgres:
  image: postgres:16-alpine
  environment:
    POSTGRES_DB: erp_matricula
    POSTGRES_USER: ${DB_USER:-matricula}
    POSTGRES_PASSWORD: ${DB_PASSWORD:-matricula}
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U ${DB_USER:-matricula} -d erp_matricula"]
    interval: 5s
    timeout: 5s
    retries: 5
  volumes: [postgres_data:/var/lib/postgresql/data]
  Comentário: "# postgres:16-alpine alinhado com o schema desenvolvido no projeto (V1__schema.sql)"

SERVICE app:
  build: .  (contexto = raiz do projeto onde o Dockerfile está)
  ports: ["8080:8080"]
  environment:
    SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/erp_matricula
    SPRING_DATASOURCE_USERNAME: ${DB_USER:-matricula}
    SPRING_DATASOURCE_PASSWORD: ${DB_PASSWORD:-matricula}
  depends_on:
    postgres:
      condition: service_healthy
  Comentário: "# SPRING_DATASOURCE_URL sobrescreve application.yml via convenção Spring Boot — sem perfis separados"
  Comentário: "# depends_on com condition: service_healthy garante que o banco está pronto antes de Flyway conectar"

VOLUMES: postgres_data (persistência entre restarts)

Comentário pedagógico no topo do arquivo (em bloco de comentários YAML com #):
"# ERP Matrícula — Docker Compose
# Sobe PostgreSQL + aplicação Spring Boot com um único comando.
#
# Como executar:
#   docker compose up          — sobe em foreground (Ctrl+C para parar)
#   docker compose up -d       — sobe em background
#   docker compose down        — para e remove containers
#   docker compose down -v     — para, remove containers E volumes (reset do banco)
#
# Variáveis de ambiente opcionais:
#   DB_USER=outro docker compose up
#   DB_PASSWORD=outra docker compose up"
